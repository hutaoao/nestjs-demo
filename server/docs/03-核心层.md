# ③ 核心层（core/）

核心层存放与业务无关的基础设施代码，包括数据库、异常处理、日志、认证等横切关注点。

## 3.1 DatabaseModule — 数据库连接

**文件**：`core/database/database.module.ts` + `database.service.ts`

```typescript
@Global()
@Module({ providers: [DatabaseService], exports: [DatabaseService] })
export class DatabaseModule {}
```

- `@Global()` 标记后，所有模块无需导入即可注入 `DatabaseService`
- `DatabaseService` 继承 `PrismaClient`，获得了所有 CRUD 方法
- 使用 Prisma 7 的驱动适配器架构：`PrismaClient → PrismaPg → pg.Pool → PostgreSQL`
- 通过 `OnModuleInit` / `OnModuleDestroy` 生命周期钩子自动管理连接

**在 Service 中注入**：
```typescript
@Injectable()
export class UsersService {
  constructor(private readonly db: DatabaseService) {}
  // db.user.create()  /  db.post.findMany() 等
}
```

**为什么叫 db 而不是 prisma？** 解耦。如果后续从 Prisma 换成其他 ORM，Controller 不受影响。

## 3.2 AllExceptionsFilter — 异常过滤器

**文件**：`core/filters/http-exception.filter.ts`

```typescript
@Catch()
export class AllExceptionsFilter implements ExceptionFilter { ... }
```

- 在 `main.ts` 中 `app.useGlobalFilters()` 全局注册
- 捕获所有异常，统一返回 `{ code, message, timestamp, path }`
- 对 `HttpException` 使用原本的状态码和消息
- 对未知异常（如数据库连接失败）记录堆栈后返回 500
- `class-validator` 的 `message` 可能是数组，自动取第一条

**未使用此 Filter 时的 404 响应**：
```json
{ "message": "Not Found", "statusCode": 404 }
```

**使用后的统一响应**：
```json
{ "code": 404, "message": "用户 #999 不存在", "timestamp": "...", "path": "/users/999" }
```

## 3.3 ApiKeyGuard — 守卫

**文件**：`core/guards/api-key.guard.ts`

Guard 决定请求是否能够到达 Controller。`ApiKeyGuard` 检查请求头 `X-API-Key`。

```typescript
@Injectable()
export class ApiKeyGuard implements CanActivate {
  canActivate(context: ExecutionContext): boolean {
    return request.header('X-API-Key') === apiKey;
  }
}
```

**三种注册方式**：

| 方式 | 代码 | 作用范围 |
|------|------|----------|
| 全局 | `app.useGlobalGuards(new ApiKeyGuard(configService))` | 所有路由 |
| 控制器 | `@UseGuards(ApiKeyGuard)` 放在 `@Controller()` 上 | 该控制器所有路由 |
| 方法 | `@UseGuards(ApiKeyGuard)` 放在 `@Get()` 上 | 单个路由 |

当前项目**没有注册此 Guard**（开发阶段默认放行），学习时可以在任意 Controller 上添加 `@UseGuards(ApiKeyGuard)` 试试效果。

## 3.4 LoggingInterceptor — 拦截器

**文件**：`core/interceptors/logging.interceptor.ts`

Interceptor 可以在请求处理前后插入额外逻辑。`LoggingInterceptor` 记录每个请求的耗时。

```typescript
@Injectable()
export class LoggingInterceptor implements NestInterceptor {
  intercept(context: ExecutionContext, next: CallHandler): Observable<unknown> {
    const { method, url } = context.switchToHttp().getRequest();
    const now = Date.now();
    return next.handle().pipe(
      tap(() => this.logger.log(`${method} ${url} → ${status} (${Date.now() - now}ms)`)),
    );
  }
}
```

已全局注册，启动后控制台会输出：
```
[Nest] LOG [HTTP] GET /users/1 → 200 (12ms)
```

## 3.5 ParseIdPipe — 管道

**文件**：`core/pipes/parse-id.pipe.ts`

Pipe 处理"数据到达 Controller 之前"的校验和转换。`ParseIdPipe` 比 NestJS 内置的 `ParseIntPipe` 多检查了 `id > 0`，并给出中文错误提示。

```typescript
@Injectable()
export class ParseIdPipe implements PipeTransform<string, number> {
  transform(value: string): number {
    const id = parseInt(value, 10);
    if (isNaN(id) || id <= 0) {
      throw new BadRequestException('ID 必须是一个正整数');
    }
    return id;
  }
}
```

**使用方式**：替换 Controller 中的 `ParseIntPipe`：
```typescript
@Get(':id')
findOne(@Param('id', new ParseIdPipe()) id: number) { ... }
```

## 3.6 CurrentUser — 自定义装饰器

**文件**：`core/decorators/current-user.decorator.ts`

`@CurrentUser()` 是 `createParamDecorator` 的典型示例，从 `request.user` 提取用户信息。

```typescript
export const CurrentUser = createParamDecorator(
  (_data: unknown, ctx: ExecutionContext) => {
    return ctx.switchToHttp().getRequest().user;
  },
);
```

**使用方式**（需配合 auth guard 先在 request 上挂载 user）：
```typescript
@Get('profile')
getProfile(@CurrentUser() user: { id: number; username: string }) {
  return user;
}
```

## 核心概念对比

| 组件 | 时机 | 作用 | 已注册 |
|------|------|------|--------|
| Guard | 请求到达前 | 权限检查，决定放行/拒绝 | ❌ |
| Pipe | 参数传入 Controller 前 | 校验数据、类型转换 | ✅ (全局) |
| Interceptor | 请求前后 | 日志、缓存、转换响应 | ✅ |
| Filter | 异常发生后 | 统一错误响应格式 | ✅ |
| Decorator | 编译时 | 定义元数据、参数提取 | — |
