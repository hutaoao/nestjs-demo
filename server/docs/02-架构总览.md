# ② 架构总览

## 目录结构

```
src/
├── main.ts                           # 入口：创建应用 + 注册全局组件
├── app.module.ts                     # 根模块：导入所有子模块 + 中间件
│
├── config/
│   └── configuration.ts              # 配置工厂：集中管理环境变量
│
├── core/                             # 核心基础设施层
│   ├── database/                     #   数据库连接（Prisma ORM）
│   ├── filters/                      #   异常过滤器（统一错误格式）
│   ├── guards/                       #   守卫（API Key 认证）
│   ├── middleware/                   #   中间件（请求计时） ← 新增
│   ├── interceptors/                 #   拦截器（日志/转换）
│   ├── pipes/                        #   管道（数据校验/转换）
│   └── decorators/                   #   自定义装饰器
│
├── shared/                           # 共享层
│   ├── dto/                          #   跨模块复用的 DTO
│   ├── interfaces/                   #   共享接口定义
│   └── shared.module.ts              #   共享模块
│
└── modules/                          # 业务层
    ├── users/                        #   用户模块（User CRUD）
    ├── posts/                        #   文章模块（Post CRUD，关联 User）
    ├── auth/                         # 🔐 认证模块（JWT + Guard + 装饰器）—— 新增
    ├── tags/                         # 🏷️ 标签模块（Tag CRUD，关联 Post）—— 新增
    ├── dynamic-config/               # 🧩 动态模块演示（forRoot）—— 新增
    ├── files/                        # 📁 文件上传模块（Multer）—— 新增
    └── demo/                         # 🔄 生命周期事件演示—— 新增
```

## 一次请求的完整生命周期（18 个概念串联）

一个 HTTP 请求到 NestJS 的完整路径：

```
客户端请求
    │
    ▼
中间件 —— 记录请求耗时               ← RequestTimerMiddleware（新增）
    │
    ▼
Guard —— 能不能进来？               ← JwtAuthGuard / RolesGuard（新增）
    │    从 ExecutionContext 拿到当前路由/方法信息
    ▼
Interceptor（前置）—— 打印请求日志   ← LoggingInterceptor（已有）
    │
    ▼
Pipe —— 参数校验 / 类型转换          ← ValidationPipe（全局）+ ParseIntPipe（路由）
    │
    ▼
Controller —— 提取参数，调用 Service ← 装饰器 @CurrentUser/@Roles/@Public
    │    用自定义装饰器提取用户信息
    ▼
Service —— 执行业务逻辑              ← 提供者（DI 注入）
    │    OnModuleInit 在服务启动时自动执行 ← 生命周期事件
    ▼
Interceptor（后置）—— 打印响应日志
    │
    ▼
Filter —— 错误时统一格式             ← AllExceptionsFilter（已有）
    │
    ▼
客户端收到响应
```

## 核心设计原则

1. **Controller 不做业务**：只负责路由映射和参数提取，业务逻辑交给 Service
2. **Service 不接触 HTTP**：不依赖 Request/Response 对象，便于测试
3. **每个模块职责单一**：一个模块只处理一个业务领域
4. **全局模块尽量少用**：只有真正"全应用共用"的（Database、Auth、Config）才用 @Global
