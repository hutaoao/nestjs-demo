# ⑤ 配置与共享（config/ + shared/）

## config/ — 集中配置管理

### 为什么不用 dotenv？

项目之前用 `import 'dotenv/config'` 直接读 `process.env`，现在改为 `@nestjs/config`：

| 方案 | 类型安全 | 必填校验 | 可测试 | 默认值 |
|------|---------|---------|--------|--------|
| `process.env.xxx` | ❌ 返回 `string\|undefined` | ❌ 手动检查 | ❌ 难 mock | ❌ 散落代码 |
| `configService.getOrThrow('xxx')` | ✅ `get<T>()` | ✅ 缺失时报错 | ✅ 可注入 mock | ✅ 集中声明 |

### 配置工厂

`config/configuration.ts` 将环境变量转为带类型的配置对象：

```typescript
export default () => ({
  port: parseInt(process.env.PORT!, 10) || 3000,
  database: {
    url: process.env.DATABASE_URL!,
  },
});
```

在 `app.module.ts` 中加载：
```typescript
ConfigModule.forRoot({ isGlobal: true, load: [configuration] })
```

### 在 Service 中使用

```typescript
constructor(private readonly configService: ConfigService) {
  const url = configService.getOrThrow<string>('DATABASE_URL');
  const port = configService.get<number>('port'); // 有默认值，不需要 getOrThrow
}
```

## shared/ — 共享代码

`shared/` 存放被多个业务模块复用的代码。

### 共享 DTO：PaginationDto

`shared/dto/pagination.dto.ts` 被 `UsersController` 和 `PostsController` 共同使用：

```typescript
export class PaginationDto {
  @Min(1) page?: number = 1;
  @Min(1) @Max(100) limit?: number = 10;
}
```

通过 `@Type(() => Number)` 自动把查询字符串 `"1"` 转成数字 `1`。

### 共享接口：ApiResponse

`shared/interfaces/api-response.interface.ts` 定义标准化的响应类型：

```typescript
interface ApiResponse<T> {
  code: number; message: string; data?: T;
  timestamp: string; path: string;
}
```

配合 `AllExceptionsFilter`，所有错误响应已自动符合此格式。

### SharedModule

```typescript
@Module({ exports: [] })
export class SharedModule {}
```

当前为空模块（共享的 DTO 和接口是纯类/类型，不需要 NestJS DI）。后续如果有共享 Service（如邮件发送、文件处理），在此注册并导出即可。

## 共享代码的使用原则

1. **DTO** 放在对应的业务模块里（`users/dto/`），只有被多个模块复用的才提到 `shared/dto/`
2. **接口/类型** 同上，先放模块内，重复了再提升
3. **Service** 需要做成 NestJS Provider 并通过 SharedModule 导出，才能被 DI 注入
