# ② 架构总览

## 目录结构

```
src/
├── main.ts                           # 入口：创建应用 + 注册全局组件
├── app.module.ts                     # 根模块：导入所有子模块
│
├── config/
│   └── configuration.ts              # 配置工厂：集中管理环境变量
│
├── core/                             # 核心基础设施层
│   ├── database/                     #   数据库连接（Prisma ORM）
│   ├── filters/                      #   异常过滤器（统一错误格式）
│   ├── guards/                       #   守卫（认证/授权）
│   ├── interceptors/                 #   拦截器（日志/转换）
│   ├── pipes/                        #   管道（数据校验/转换）
│   └── decorators/                   #   自定义装饰器
│
├── shared/                           # 共享层
│   ├── dto/                          #   跨模块复用的 DTO
│   ├── interfaces/                   #   共享接口定义
│   └── shared.module.ts              #   共享模块
│
└── modules/                          # 业务层
    ├── users/                        #   用户模块（User CRUD）
    │   ├── users.module.ts
    │   ├── users.controller.ts
    │   ├── users.service.ts
    │   ├── dto/                      #     CreateUserDto, UpdateUserDto
    │   └── entities/                 #     占位（Prisma 不需要实体文件）
    └── posts/                        #   文章模块（Post CRUD，关联 User）
        ├── posts.module.ts
        ├── posts.controller.ts
        ├── posts.service.ts
        └── dto/                      #     CreatePostDto, UpdatePostDto
```

## 一次请求的完整生命周期

一个 HTTP 请求到 NestJS 的完整路径：

```
客户端请求
    │
    ▼
Guard（守卫） —— 能不能进来？         ← ApiKeyGuard（可选注册）
    │
    ▼
Interceptor（拦截器·前置）—— 打印请求日志  ← LoggingInterceptor（已注册）
    │
    ▼
Pipe（管道） —— 参数校验 / 转换        ← ValidationPipe（全局）+ ParseIntPipe（路由）
    │
    ▼
Controller —— 提取参数，调用 Service   ← UsersController
    │
    ▼
Service —— 执行业务逻辑 + 数据库操作   ← UsersService → DatabaseService → Prisma
    │
    ▼
Interceptor（拦截器·后置）—— 打印响应日志
    │
    ▼
Filter（异常过滤器） —— 错误时统一格式   ← AllExceptionsFilter（已注册）
    │
    ▼
客户端收到响应
```

## 核心设计原则

1. **Controller 不做业务**：只负责路由映射和参数提取，业务逻辑交给 Service
2. **Service 不接触 HTTP**：不依赖 Request/Response 对象，便于测试
3. **每个模块职责单一**：一个模块只处理一个业务领域（User / Post）
4. **全局模块尽量少用**：只有真正"全应用共用"的（Database、Config）才用 @Global
